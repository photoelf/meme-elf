# meme-elf Roadmap

## Project Shape

`meme-elf` is now a working static browser meme editor with production alpha hosting on `pages.dev`.

The product has three long-term usage modes:
1. `Quick meme`
2. `Image edit++`
3. `Template meme`

The first mode is already live and must remain fast.

## Delivery Principle

Optimize for:
- preserving the fast `Ctrl+V -> text -> Ctrl+C` flow
- adding advanced editing through progressive disclosure
- staying static-host compatible
- keeping operations and deployment simple

### Working alpha baseline

Current production baseline includes:
- clipboard paste and file upload
- multiple editable text layers
- move, resize, rotate, reorder
- image layers with remove, rotate, flip, reorder, and top-of-stack insertion
- pre-insert modal for UI-driven imports with crop, rotate, and flip
- scene crop with draggable crop box, resize handles, start-outside-image behavior, and scene image-stack rotate / flip actions
- canvas expansion on any side
- transparent, solid, sampled-edge, and average-border fill for expanded area
- bounds presets for equal margin, caption space, and square canvas
- advanced import from file or clipboard
- inline canvas text editing with multiline support and live re-fit after line removal
- preview zoom with `Fit` as the default load mode, plus `1:1` and middle-mouse pan
- undo / redo on toolbar and physical-key `Ctrl/Cmd+Z` shortcuts
- clipboard copy and PNG download
- production deployment on Cloudflare Pages

### Epic 6C: Effects and Watermarks

Deliverables:
- common image filter stack
- meme degradation effects
- watermark tools
- blend mode foundations where useful

Suggested 5-hour batches:
- `6C-1`: filter state model and non-destructive preview pipeline
- `6C-2`: brightness, contrast, saturation, hue, grayscale, sepia, invert
- `6C-3`: blur, sharpen, threshold, pixelate, noise
- `6C-4`: posterize and JPEG degradation effects
- `6C-5`: watermark generator modes and controls

Implemented so far:
- `6C-1`
- `6C-2`
- `6C-3`
- `6C-4`
- `6C-5`

Known alpha behavior:
- with `Apply to text` disabled, effects apply to base image plus image layers while text stays above the filtered image stack
- with `Apply to text` enabled, text joins the filtered scene pass and can render behind imported image layers; this is accepted for the current alpha because the default remains `off`
- `Adjustments` are fixed-order scene controls, while `Effects` are a reorderable stack whose visible order is also the processing order
- watermark controls live in their own `Watermark` inspector tab, with defaults anchored around an enabled lower-left `создано в программе meme-elf` corner mark at `12px` and optional corner/tile placement controls

Recommended execution order for `6C`:
- `6C-1`
- `6C-2`
- `6C-3`
- `6C-4`
- `6C-5`

### Epic 7C: Advanced Retouch Research

Deliverables:
- evaluate clone stamp feasibility
- evaluate healing or patch feasibility
- decide what stays custom and what requires a library

Suggested 5-hour batches:
- `7C-1`: shipped local-only narrow clone stamp on custom stack
- `7C-2`: healing or patch feasibility research with candidate library matrix
- `7C-3`: architecture decision and adoption recommendation

Implemented so far:
- `7C-1`
- `7C-2`

Research outcome:
- simple clone stamp is feasible on the current raster stack when sampling stays one-target-at-a-time and reads from a frozen source snapshot during each stroke; the shipped slice currently lives behind the local-only `Experimental` tab
- healing, seamless patch, or content-aware repair are not good custom candidates for the current alpha-plus scope
- no dependency adoption is justified yet; revisit only if a later milestone explicitly needs healing-quality retouch

## Technology Direction

Default approach:
- keep using React + TypeScript + browser canvas
- evaluate new libraries only when they unlock a clear feature cluster

Strong candidates for custom implementation:
- crop
- canvas expansion
- image transforms
- image layering
- draw and erase
- rectangular selection
- common filters
- watermarking

Likely evaluation areas before adopting dependencies:
- advanced perspective tooling
- advanced retouch and inpaint-like workflows
- more complex filter pipelines
